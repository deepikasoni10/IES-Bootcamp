RDE Filing Lifecycle — Complete API Flow
Step API Action Who → Who Type What Happens State After
0 Setup / Register Infosys → DeDi + Cred Service SETUP Register in DeDi directory. Get Verifiable Credential from credential service. registered
1a /search BAP → Gateway → BPP SYNC Find SERC endpoint. Intent: ARR_petition, regulatory_filing. searching
1b /on_search BPP → BAP /callback/on_search ASYNC SERC catalog returned with supported filing types, schema refs (ies:DatasetPayload:ARR:v1.0). ready
2a /init (Draft) BAP → Gateway → BPP SYNC ACK Send draft filing: DatasetPayload + payload_hash + payload_url + envelope. BPP returns HTTP 200 ACK immediately. draft_submitted
2b /on_init BPP → BAP /callback/on_init ASYNC ValidationReport: schema_check, hash_check, completeness_check, observations with field-level warnings. draft_validated
3a /confirm (Formal) BAP → Gateway → BPP SYNC ACK Formal submission: W3C Verifiable Credential + Ed25519 digital_signature + payload_hash + provenance chain. BPP verifies credential + hash. formally_submitted
3b /on_confirm BPP → BAP /callback/on_confirm ASYNC Signed RegulatoryFilingReceipt: filing_id, status (accepted/rejected), accepted_payload_hash, disclosure_catalog_url, SERC Ed25519 proof. accepted / rejected
4a /status BAP → Gateway → BPP SYNC Check filing state by filing_id: "SERC/ARR/2026-27/MH/001" —
4b /on_status BPP → BAP /callback/on_status ASYNC filing_status: "disclosed" + disclosure catalog_url + content_hash for verification. disclosed
5 Public Disclosure SERC → DeDi Catalog PUBLIC Filing published as static JSON on DeDi. Anyone can: discover → fetch catalog → download data → verify SHA-256 hash → verify SERC signature. publicly_disclosed

All 10 Components — What to Build at Bootcamp
1. CSV / Excel Parser  P0 · 1-2 hrs
Input: DISCOM CSV/Excel exports (.csv, .xlsx)
Output: Parsed tabular data in memory (dict/dataframe)
Libraries: pandas, openpyxl (Python)
Handle: Multiple sheets, merged cells, varying date formats, missing values
2. Schema Mapper  P0 · 2-3 hrs
Input: Parsed tabular data from Component 1
Output: IES DatasetPayload JSON-LD document
Config: YAML mapping file per DISCOM (column → IES field)
Must include: @context, @type, financial_year, filing_category, discom_id
3. Content Hasher (SHA-256)  P0 · 30 min
Input: DatasetPayload JSON dict
Output: "sha256:hexstring"
canonical = json.dumps(payload,
  sort_keys=True, separators=(',',':'))
return "sha256:" + hashlib.sha256(
  canonical.encode('utf-8')).hexdigest()
⚠ Confirm canonicalization method with IES team on Day 1
4. CommonEnvelope Builder  P0 · 1 hr
Input: DatasetPayload + hash + issuer identity
Output: Complete IES Filing object
Fields: schema_version, content_hash, issuer_id, created_at, provenance (links to prior draft)
5. Beckn Message Builder  WAS MISSING — NOW ADDED  P0 · 1-2 hrs
Input: Action type (search/init/confirm/status), filing payload, credentials
Output: Complete Beckn JSON { context + message }
Handles: UUID gen for transaction_id + message_id, timestamps, BAP/BPP IDs, domain
Rule: Same transaction_id across entire lifecycle (init → confirm → status)
domain: "ies:regulatory" (confirm on Day 1)
6. Receipt Processor  P1 · 1 hr
Input: Signed Receipt from SERC via on_confirm
Steps:
1. Verify SERC's Ed25519 signature on receipt
2. Check: accepted_payload_hash == submitted hash
3. Store receipt + provenance chain in DB
4. If accepted: note disclosure_catalog_url
7. Credential Manager  P1 · 1-2 hrs
Standard: W3C Verifiable Credentials Data Model v1.1
Algorithm: Ed25519Signature2020
Steps:
1. Request VC from sandbox credential service (DISCOM identity)
2. Store VC locally
3. Attach VC to /confirm request
4. Sign payload hash with private key
⚠ Get credential service API docs on Day 1 morning
8. Async Callback Server  ENDPOINTS NOW EXPLICIT  P0 · 1-2 hrs
Framework: Python FastAPI or Flask
Endpoints (all POST):
POST /callback/on_search → store SERC catalog
POST /callback/on_init → parse + store ValidationReport
POST /callback/on_confirm → parse + verify signed Receipt
POST /callback/on_status → parse Disclosure info
⚠ Must be reachable from Beckn network — register BAP callback URL in ONIX sandbox on Day 1
9. Filing State Machine  P1 · 1 hr
Storage: In-memory dict keyed by transaction_id (bootcamp); PostgreSQL (production)
preparing → /init sent → draft_submitted
draft_submitted → on_init rcvd → draft_validated
draft_validated → /confirm sent → formally_submitted
formally_submitted → on_confirm rcvd → accepted / rejected
accepted → DeDi publish → disclosed
10. JSON Schema Validator  P0 · 30 min
Library: jsonschema (Python)
Schemas needed from IES GitHub:
• CommonEnvelope JSON schema
• Filing JSON schema
• DatasetPayload (ARR) JSON schema
• Receipt JSON schema
When: Validate before /init AND before /confirm — catch locally
⚠ Get actual .json schema files from IES GitHub on Day 1 first thing

What Was Fixed in This Architecture
❌ Items Missing in Original Diagram → ✅ Fixed in This Version
❌ Fix 1: Beckn Message Builder was Missing from RDE Service
Problem: Original diagram showed CSV Parser → Mapper → Hasher → Envelope Builder — but how does the Beckn-formatted JSON get constructed? The component that builds { context + message }, generates UUIDs, and routes to ONIX was completely absent.
Fix: Added Component 5 (Beckn Message Builder) explicitly in RDE Service as a distinct step in the pipeline.
❌ Fix 2: Callback Server Endpoints Were Not Listed
Problem: "Async Callback Server" was shown as a black box — the 4 specific POST endpoints were not listed, making implementation unclear.
Fix: All 4 endpoints now explicitly shown:
POST /callback/on_search
POST /callback/on_init → parse ValidationReport
POST /callback/on_confirm → parse signed Receipt
POST /callback/on_status → parse Disclosure info
❌ Fix 3: DeDi Public Disclosure Flow Was Missing
Problem: DeDi was shown as just a storage layer — the Public Disclosure step (accepted filing → SERC publishes to DeDi catalog → publicly accessible) was not shown anywhere.
Fix: Added "Public Disclosures" in DeDi catalog (Layer 6) and the full 5th step of the lifecycle: accepted → SERC publishes static JSON → anyone can discover + verify hash + verify SERC signature.
⚠ Fix 4: TI Service + EDX Service Scope Not Clarified
Problem: TI (Tariff Intelligence) and EDX (Energy Digest/Telemetry Exchange) Services were shown at same priority as RDE Service — creating confusion about what to build in 3 days.
Fix: Both marked as "Post-Bootcamp" with dashed borders. Bootcamp scope = RDE only.
⚠ Fix 5: Keycloak + API Gateway Shown as Bootcamp Items
Problem: Both Keycloak IAM and API Gateway were shown without any note that they are production concerns — implying the team should build these during bootcamp.
Fix: Both labeled "Production Concern — Sandbox has own auth" so team focuses on RDE components instead.
⚠ Fix 6: Consumer App + Admin Dashboard Not Marked as Post-Bootcamp
Problem: Consumer App (EDX View) and Admin Dashboard (TI Admin) appeared in Layer 1 without any priority distinction.
Fix: Both marked as "Post-Bootcamp" — only DISCOM Portal (RDE Filing) and Postman API Client are in bootcamp scope.
